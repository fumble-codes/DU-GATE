Flow

Auth Flows Every pathway through the identity layer

Email + Password Login

POST /auth/login — standard credential-based authentication via Supabase GoTrue.

Google OAuth (Authorization Code + PKCE)

signInWithOAuth({ provider:"google" }) — three-party handshake with PKCE verification.

Cookie Creation

After exchangeCodeForSession, Supabase sets sb-*-auth-token and sb-*-refresh-token on the response. These are HttpOnly, Secure, SameSite=Lax cookies.

Server Read

proxy.ts reads cookies via request.cookies.getAll(), passes to createServerClient() which calls getSession() → GoTrue validates the JWT.

Client Sync

createBrowserClient() mirrors cookies; onAuthStateChange fires when tokens rotate. The AuthProvider listens to this and updates user / role state.

Auto-Profile Trigger: handle_new_user()

Fires AFTER INSERT ON auth.users. Creates the corresponding profiles row. SECURITY DEFINER — runs as owner, bypasses RLS. SET search_path = public ensures the table is found. 

-- database/migrations/001_profiles.sql
CREATE OR REPLACE FUNCTION handle_new_user()
RETURNS trigger
SECURITY DEFINER
SET search_path = public
AS $$
BEGIN
  INSERT INTO public.profiles (id, username, avatar_url)
  VALUES (
    NEW.id,
    COALESCE(
      NEW.raw_user_meta_data->>'username',
      NEW.raw_user_meta_data->>'full_name',
      NEW.raw_user_meta_data->>'name',
      ''
    ),
    NEW.raw_user_meta_data->>'avatar_url'
  );
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
✅ Fixed in v003
SET search_path = public — the root cause of the Database error saving new user. Without it, the function couldn't find profiles.
📋 Must run before OAuth works
Apply database/migrations/003_fix_profile_trigger.sql in Supabase SQL Editor.
RBAC

Role Propagation How a USER becomes ADMIN (and who checks)

Proxy Gate

On every /admin/:path* request, queries profiles.role. USER → /403. ADMIN → allow.

verifyAdmin()

Server component guard in (admin)/layout.tsx. Double-checks before rendering admin shell. Defense in depth.

AuthProvider

Client-side React context. Provides user, role, signOut. Drives UI: different nav for USER vs ADMIN.

Security

Threat Model Every attack surface and its mitigation

🔓 Session hijacking / XSS token theft

JWT in HttpOnly cookies (not localStorage). XSS cannot read them. Dashboard migrated off getSession() (localStorage) to useAuth().

🔄 CSRF

SameSite=Lax cookies. State-changing calls go through Supabase signed requests.

↗️ Open redirect

Supabase registers only whitelisted Redirect URLs; redirectTo is origin-bound.

⬆️ Privilege escalation (USER→ADMIN)

role only settable via DB / profiles RLS. Double-checked in proxy + guard (defense in depth).

🔑 Service-role key leak

Regenerated. Never exposed to client. RLS still bounds anon access.

💥 Brute force / rate-limit

Rely on Supabase rate-limiting. Consider CAPTCHA on signup for Phase 8 hardening.

🐞 DB error saving new user (Google OAuth)

Trigger lacked search_path and metadata fallback. Fixed by 003_fix_profile_trigger.sql. Must run on DB.

📝 Malformed / expired JWT

GoTrue validates signature, expiry, and claims. Treated as null session if invalid.
