Why Each Table Exists

Purpose of every table, what it owns, and how it connects to others
subjects entry point for content hierarchy
Why it exists Groups questions by academic subject (Accountancy, Economics, English, etc.). Every question must belong to a subject indirectly through a chapter.
How it connects 1:N → chapters via chapters.subject_id. Also referenced by mock_tests via mock_tests.subject_id so mocks are scoped to a subject.
chapters syllabus breakdown within a subject
Why it exists Breaks a subject into teachable units (e.g. "Partnership", "Ratio Analysis"). Students practice by chapter; questions are organized by chapter.
How it connects N:1 ← subjects via subject_id · 1:N → questions via questions.chapter_id. ON DELETE RESTRICT prevents deleting a chapter that still has questions.
questions the core — every question lives exactly once
Why it exists The single source of truth. No feature duplicates a question — practice, mocks, revision, AI analysis all reference this table. Stores the body, explanation, difficulty, type, and status.
How it connects N:1 ← chapters · N:0..1 ← question_groups (optional, SET NULL) · 1:N → options · 1:N → question_concepts · 1:N → question_attachments · 1:N → mock_questions.
options answer choices, normalized out of the question row
Why it exists Questions never store options inline. Normalizing options allows flexible expansion (4 options, 5 options, future types), independent ordering, and marking the correct one via is_correct.
How it connects N:1 ← questions via question_id. ON DELETE CASCADE — when a question is removed, its options go with it. Each option has a unique display_order per question.
concepts cross-cutting learning labels, independent of chapters
Why it exists Concepts (like "Goodwill", "Depreciation") cut across chapters and subjects. A question can cover multiple concepts; a concept can span thousands of questions. This enables weak-concept analysis and targeted revision.
How it connects M:N with questions via the question_concepts junction table. question_id CASCADE (question removal cleans up links), concept_id RESTRICT (concepts are protected).
question_groups shared passages, case studies, and common content
Why it exists A single passage or table shared by multiple questions (case studies, reading comprehension). Without this, the same passage would be duplicated in every question body. Normal MCQs leave group_id as NULL.
How it connects 1:N → questions via questions.question_group_id (optional). ON DELETE SET NULL — deleting a group leaves the questions intact, just ungrouped.
question_attachments media assets linked to a question
Why it exists Questions often include images, graphs, tables, or diagrams. Files are stored in Supabase Storage; the database stores only the storage_path. This keeps the question row lightweight.
How it connects N:1 ← questions via question_id. ON DELETE CASCADE — attachments are part of the question and go with it.
mock_tests test metadata — never duplicates questions
Why it exists Defines a mock test: title, duration, marking scheme, publication status. Questions are not stored here — they're linked via the junction table, enabling one question to appear in many mocks.
How it connects N:1 ← subjects via subject_id. M:N with questions via mock_questions. mock_test_id CASCADE (delete a mock, its links go too), question_id RESTRICT (protects the question bank).
question_attempts per-user, per-question attempt record
Why it exists Tracks every question a user has answered: what they selected, whether it was correct, and how long they spent. Powers analytics, weak-concept detection, and progress tracking without touching the question bank.
How it connects N:1 ← auth.users via user_id · N:1 ← questions via question_id · N:1 ← options via selected_option_id (nullable). All CASCADE.
mock_attempts per-user mock test session record
Why it exists Records a completed mock attempt: score, breakdown of correct/incorrect/skipped, and timing. Separates attempt data from the immutable mock definition and question bank.
How it connects N:1 ← auth.users via user_id · N:1 ← mock_tests via mock_test_id. Both CASCADE.

Row-Level Security

Two security domains — public read, owner-scoped write
Content Domain — Public Read
subjects chapters concepts question_groups questions options question_concepts question_attachments mock_tests mock_questions
SELECTUSING (true)— anyone can read
ALLUSING (is_admin(auth.uid()))— admins can write
is_admin() is a SECURITY DEFINER function that checks profiles.role = 'ADMIN'. It bypasses RLS to avoid infinite recursion.
User Data Domain — Owner Scoped
question_attempts mock_attempts
SELECTuser_id = auth.uid() OR is_admin()
INSERTuser_id = auth.uid()— only self-insert
Why? The app uses the anon key for both browser and server Supabase clients. Without RLS, the anon role has full database access. RLS is the only security boundary.
